# 2026-05-28 — kashi 0.8.0 security & hardening audit

**Scope**: full re-walk of the four parser surfaces after the 0.7.x
import-formats track closed (PSF, BDF, PCF, plus the lib.cyr
registration and attach paths). The 0.6.0 P(-1) audit covered the
post-0.1.0 surface; this audit covers the 0.7.0/0.7.1/0.7.2
additions and re-examines the existing surface against the
2020–2026 font-parser CVE corpus.

**Method**:
1. CVE research (libXfont, FreeType, X.Org advisories 2006–2025;
   FontForge 2024–2025) — extracted a 17-point checklist of attack
   patterns (`count * size` overflow, format-byte unknown bits,
   stale state after parse failure, etc.).
2. File-by-file walk against the checklist for `src/font_psf.cyr`,
   `src/font_bdf.cyr`, `src/font_pcf.cyr`, and `src/lib.cyr`.
3. Fixes applied, regression tests added (one per finding), full
   suite re-run.

**Result**: 9 findings, 0 exploitable. Severity skews low —
hardening / defense-in-depth — with one medium finding (F9, stale
state after a failed attach). All fixed in this cut.

## Reference: CVE patterns drawn on

| CVE | Pattern |
|---|---|
| **CVE-2006-3467** (libXfont PCF) | `count * sizeof(...)` overflow before allocation |
| **CVE-2008-0006** (libXfont PCF) | TOC `nTables` / `(offset, size)` not validated; **duplicate-type** TOC entries (→ F8) |
| **CVE-2014-0209 / 0210 / 0211** (libXfont) | Integer overflow in metadata size calc; trusted reply lengths |
| **CVE-2015-1802** (libXfont BDF) | Property count not bounded → overflow on multiply |
| **CVE-2015-1803** (libXfont BDF) | **Stale pointer / state after parse failure** (→ F9) |
| **CVE-2015-1804** (libXfont BDF) | Metric values truncated from int32 → int16 |
| **CVE-2017-13720 / 13722** (libXfont2) | OOB read from unbounded string-table walk |
| **CVE-2025-27363** (FreeType) | Sign-extension hygiene + alloc-size validation (RCE, exploited ITW) |

The two highest-yield rules from the corpus:
1. **Every `a * b` feeding an allocation must be done in i64 with a
   bound check.**
2. **Every offset+size must be validated against file size with the
   add done in i64.**

Both rules audited clean across all four kashi parsers — Cyrius's
default i64 width means the multiplies are 64-bit-natural, and the
explicit bounds (`count ≤ KASHI_PSF_MAX_COUNT`, etc.) cap the inputs
to multiplies an additional time.

## Findings

### F1 — PSF1 unknown mode bits not rejected · Low · Fixed

**File**: `src/font_psf.cyr` (`_kashi_psf1_parse`).

PSF1 defines mode bits 0–2 (MODE512 | MODEHASTAB | MODESEQ). Bits
3–7 were ignored. An adversarial PSF1 with junk in the high mode
bits was accepted.

**Risk**: no memory-safety implication (the code only reads the
defined bits), but a known-malformed input was treated as valid —
defense-in-depth violation.

**Fix**: `if ((mode & 0xF8) != 0) { return KASHI_EFORMAT; }`.

**Regression**: `test_audit_f1_psf1_unknown_mode` in `src/test.cyr`
asserts mode `0x10` is rejected, mode `0x07` (all defined bits) is
accepted.

### F2 — PSF2 unknown flags bits not rejected · Low · Fixed

**File**: `src/font_psf.cyr` (`_kashi_psf2_parse`).

PSF2 defines only flag bit 0 (HAS_UNICODE). Bits 1–31 were ignored.

**Risk**: same defense-in-depth shape as F1.

**Fix**: `if ((flags & 0xFFFFFFFE) != 0) { return KASHI_EFORMAT; }`.

**Regression**: `test_audit_f2_psf2_unknown_flags`.

### F3 — BDF integer parser silently truncated at 10 digits · Low · Fixed

**File**: `src/font_bdf.cyr` (`_kashi_bdf_int`).

The parser capped digit consumption at `KASHI_BDF_INT_MAX_DIGITS`
(= 10) to avoid i64 overflow in the accumulator, but on hitting the
cap it returned **success** with the truncated value, leaving
leftover digits for the next field's parse. A field like
`CHARS 99999999999` (11 digits) was being read as `9999999999`
(10 digits, ~10^10) — still inside the `KASHI_BDF_MAX_COUNT`
(65,536) check, but the leftover `9` corrupted the next field's
parse.

**Risk**: fail-quiet hygiene violation. The downstream
`KASHI_BDF_MAX_COUNT` check caught the truncated value, so no
exploit was possible — but the silent-truncate pattern is the same
shape as CVE-2015-1804's metric truncation.

**Fix**: on hitting `KASHI_BDF_INT_MAX_DIGITS`, set an `overflow`
flag and return failure (`return 0`).

**Regression**: `test_audit_f3_bdf_int_overflow` — `CHARS 99999999999`
now → `KASHI_BDF_EFORMAT`.

### F4 — Dead code in PCF compressed-metrics path · Trivial · Fixed

**File**: `src/font_pcf.cyr` (`_kashi_pcf_parse_metrics`).

Two consecutive assignments to `ref_left` — the first using a now-
removed `_kashi_pcf_loadi8` helper, the second the canonical
`load8 - 0x80` form. The first was dead.

**Risk**: none — purely cosmetic. Flagged because dead code is a
code-smell signal that the author had two approaches in flight.

**Fix**: removed the dead assignment and the now-unused
`_kashi_pcf_loadi8` helper.

### F5 / F6 / F7 — PCF format byte unknown bits not rejected · Low · Fixed

**Files**: `src/font_pcf.cyr` (`_kashi_pcf_parse_metrics`,
`_kashi_pcf_parse_bitmaps`, `_kashi_pcf_parse_encodings`).

The PCF format word's bits 0–5 are layout/order flags; bit 8 is
`PCF_COMPRESSED` (METRICS only). All other bits are reserved and
should be 0. Each per-table parser was masking out the bits it
needed but accepting unknown bits silently.

**Risk**: defense-in-depth violation; same family as F1/F2 but for
a different format. CVE-2008-0006 cited TOC field handling without
range validation as one of the libXfont PCF issues.

**Fix**: added `PCF_FMT_VALID_METRICS = 0x13F` and
`PCF_FMT_VALID_OTHER = 0x3F` constants; each parser now rejects with
`KASHI_PCF_EFORMAT` if `(fmt & VALID) != fmt`.

**Regressions**: `test_audit_f5_pcf_metrics_unknown_bits`,
`test_audit_f6_pcf_bitmaps_unknown_bits`,
`test_audit_f7_pcf_encodings_unknown_bits` — each patches a TOC
entry's format with an undefined bit and confirms rejection.

### F8 — PCF duplicate TOC entries silently used first · Low · Fixed

**File**: `src/font_pcf.cyr` (`kashi_pcf_parse_header`).

`_kashi_pcf_find_table` returns on the first match. A malformed PCF
with two `PCF_TYPE_METRICS` (or any other type) entries would have
the second silently ignored.

**Risk**: CVE-2008-0006 cited libXfont's behavior of taking the
**last** duplicate without validation as the actual issue. kashi
taking the first is safer, but rejecting upfront is cleaner — a
PCF with duplicate type entries is unambiguously malformed.

**Fix**: new `_kashi_pcf_has_dup_tables(buf, table_count)` runs once
after TOC bounds are validated; O(N²) over `N ≤ 64` entries.

**Regression**: `test_audit_f8_pcf_duplicate_toc` — patches the
ENCODINGS entry's type to METRICS, confirms `KASHI_PCF_EFORMAT`.

### F9 — `kashi_attach_unicode_table` left font with no map on failed attach · Medium · Fixed

**File**: `src/lib.cyr` (`kashi_attach_unicode_table`).

The pre-fix flow:

1. Clear `RT_UMAP` / `RT_UCOUNT` / `RT_SEQS` / `RT_SEQCOUNT` /
   `RT_SEQDATA` on the runtime record.
2. Call `_kashi_build_umap` and `_kashi_build_useqs` (which only
   write fresh values if they find entries).
3. If both pass produced zero entries, return `KASHI_EFORMAT`.

But step 3 returned the error **after** step 1 had already wiped
the existing map. A consumer attaching a malformed table on a font
that *previously had a valid map* would get `KASHI_EFORMAT`, see
the error, leave the font alone — only to find that subsequent
codepoint lookups silently returned blanks. The font's Unicode map
had been quietly stripped.

This is **CVE-2015-1803's stale-state-after-parse-error pattern**:
the parser cleaned up partially on the error path. Not exploitable
in kashi (no memory corruption — just observable inconsistency),
but it violates the consumer contract: a returned error should
imply "no state change."

**Fix**: snapshot the previous values before clearing, restore them
on failure:

```cyrius
var prev_umap = load64(rec + RT_UMAP);
var prev_ucount = ...;
# ...
store64(rec + RT_UMAP, 0);   # clear
# ...
_kashi_build_umap(...);
_kashi_build_useqs(...);
if (load64(rec + RT_UCOUNT) == 0) {
    if (load64(rec + RT_SEQCOUNT) == 0) {
        store64(rec + RT_UMAP, prev_umap);    # restore
        # ...
        return 0 - KASHI_EFORMAT;
    }
}
return KASHI_OK;
```

**Regression**: `test_audit_f9_attach_stale_state` — loads a PSF
with an inline Unicode table (gives a starting map), attempts to
attach an empty binary table (just the PSF1 terminator), confirms
the attach returns negative AND the original codepoint is still
mapped post-failure.

## Items considered and found clean

The remaining 8 checklist items came up clean on walk-through.
Notes on the ones worth recording explicitly:

- **`count * charsize` overflow guards** (PSF/BDF/PCF, lib.cyr) —
  already in place since 0.6.0 audit F1 for `kashi_register_font`;
  re-confirmed for all four import paths. Worst case:
  `65536 * 128 = 8 MiB`, well inside i64.
- **TOC `offset + size > len`** — checked at parse_header time in
  PCF. Each table also re-checks against its own bounds before any
  indexed read. The find_table helper itself does
  `if (entry + KASHI_PCF_TOC_STRIDE > len) { return 0; }` per iter.
- **PCF metric signedness** — compressed metrics use the `byte -
  0x80` convention (correctly handled); uncompressed use
  `_kashi_pcf_loadi16` with explicit sign extension. The uniform-
  metrics check compares signed-i64 values, so width/height stay
  consistent across the file.
- **PCF glyph bitmap size** — computed in i64 from bounded inputs
  (`height ≤ 32`, `pcf_row_bytes ≤ 8` since `glyph_pad ≤ 8`); the
  per-glyph `src + total > data_end` check is the canonical bounds
  guard.
- **Unicode-table walk termination** — `kashi_psf_uni_token`
  guarantees `pos` strictly advances or returns `END`. The umap
  build loops use `while (pos < len)` directly.
- **String-table reads with NUL bounds** — kashi has no string-table
  reads (PCF's `PCF_PROPERTIES` is among the **ignored** tables;
  ADR 0009).
- **Type confusion in load32 sign extension** — `_kashi_pcf_load32`
  is OR-combined from `load8` u8 values, so the result is always
  in `[0, 0xFFFFFFFF]` (no sign extension). The downstream bounds
  checks (`if (count > MAX)`) work whether the value is treated as
  signed or unsigned.
- **`KASHI_PSF_MAX_COUNT` / `KASHI_BDF_MAX_COUNT` /
  `KASHI_PCF_MAX_COUNT`** — all 65,536. Consistent across all three
  parsers; matches the libXfont historical bound.

## Items dropped from the roadmap

The 0.7.2 roadmap also mentioned "larger table-walk caps" (the PSF
side). On re-examination in ADR 0010 (0.7.2) this was found to be a
non-issue — the table walk is buffer-bounded with no infinite-loop
risk. No change needed; recorded here for the audit trail.
