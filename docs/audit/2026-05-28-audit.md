## Bounds analysis ‚Äî re-verification

The original 0.1.0 audit established that every caller-facing core
accessor is safe across the full `i64` domain. Re-walked the post-0.1.0
surface against the same invariants:

- **`kashi_glyph_ptr/row/row_byte/encoded/count/stride`** (`font_data.cyr`):
  ranges widened to `0x20..0xFF` (224 glyphs); buffer sizes grew to match
  (`kashi_font16[3584]`, `kashi_font8[1792]`, `kashi_font9_16[7168]`).
  Worst-case offset:
  - VGA 8√ó16: `(0xFF - 0x20) * 16 + 15 = 3535` ‚â§ 3583 ‚úì
  - CGA 8√ó8: `(0xFF - 0x20) * 8 + 7 = 1759` ‚â§ 1791 ‚úì
  - VGA 9√ó16: `(0xFF - 0x20) * 16 * 2 + 16*2 - 1 = 7167` ‚â§ 7167 ‚úì
  Stride-aware `kashi_glyph_row`: inline fast-path for stride 1 (built-ins
  except 9√ó16); branch on font_id 2 for stride 2. No function call on the
  hot path.
- **VGA 9√ó16 derivation loop** (`kashi_font_init`): iterates `dch` over
  `[FIRST, LAST]` (224 values), `dr` over `[0, 16)`. Max source read:
  `&kashi_font16 + (LAST-FIRST)*16 + 15 = 3583`; max dest store:
  `&kashi_font9_16 + (LAST-FIRST)*32 + 31 = 7167`. Both ‚â§ buffer size.
- **PSF parser** (`font_psf.cyr`): every header field validated before
  the data length check; `count` bounded by `KASHI_PSF_MAX_COUNT` (65536),
  `charsize` by `height * stride` ‚â§ `32 * 4 = 128` for valid widths,
  giving `glyph_bytes` ‚â§ 8 MiB (fits i64 with room).
  Fuzz: 4 000 rounds of random/truncated/mutated buffers (width 1‚Äì32
  randomized post-0.5.0), no crashes.
- **`kashi_psf_uni_token`**: always bounds-checked against `end`; UTF-8
  decode rejects truncated and (now per F2) out-of-range / surrogate
  codepoints.

## Findings

### F1 (Med/High) ‚Äî `kashi_register_font`: unbounded `glyph_count`

`nbytes = glyph_count * charsize` was computed without an upper bound on
`glyph_count`. A consumer passing `glyph_count = 0x7FFF_FFFF_FFFF_FFFF`
would overflow i64, produce a small `nbytes`, allocate a small buffer,
then `memcpy(owned, glyph_data, nbytes)` would walk off the *source*
buffer the caller supplied. The PSF parse path is bounded
(`KASHI_PSF_MAX_COUNT = 65536`), but the direct `kashi_register_font` API
was not.

**Fix**: add `if (glyph_count > KASHI_PSF_MAX_COUNT) { return 0 -
KASHI_EINVAL; }` to match the PSF parser's bound. With `width ‚â§ 32`,
`height ‚â§ 32`, `glyph_count ‚â§ 65536` ‚Üí `nbytes ‚â§ 8 MiB`. No overflow.

**Test**: `test_register_huge_count` ‚Äî bound rejection at `MAX+1`, at
`0x7FFFFFFF`, and a sanity check that exactly-at-the-cap is still
accepted.

### F2 (Low) ‚Äî UTF-8 decoder accepts out-of-range codepoints

`kashi_psf_uni_token` (PSF2 path) decoded continuation bytes into `cp`
without rejecting:
- Codepoints above `U+10FFFF` (the Unicode max; produced by lead bytes
  ‚â• `0xF4` with high enough continuation bytes).
- The UTF-16 surrogate range `U+D800..U+DFFF` (invalid in UTF-8).

No memory unsafety ‚Äî `cp` is just a u64 used as a map key ‚Äî but lets
garbage into the codepoint‚Üíglyph map. Spec-correct UTF-8 should reject
these.

**Fix**: after assembling the codepoint and before emitting
`KASHI_TOK_CP`, return `KASHI_TOK_END` if `cp > 0x10FFFF` or
`cp >= 0xD800 && cp <= 0xDFFF`. Walk terminates cleanly; downstream map
build skips. Bounded cost (two compares per multi-byte char).

**Test**: `test_uni_token_strict_utf8` ‚Äî explicit byte sequences for
`U+110000` (rejected), `U+D800` (rejected), `U+DFFF` (rejected), `U+D7FF`
(accepted ‚Äî just below surrogates), `U+10FFFF` (accepted ‚Äî max valid).

### F3 (Doc) ‚Äî Stale comments

Several scope/version notes had not kept pace:
- `font_psf.cyr` header "Scope (M1, 0.2.0)" with `width 1..8 only` and
  "Unicode table presence is recorded, not yet mapped" ‚Äî both have evolved.
- `KASHI_PH_WIDTH` doc said `1..8` (now 1..32).
- `kashi_psf_parse` doc said `48-byte` out struct (now 56).
- `lib.cyr` header "Skeleton only as of 0.1.0" / "stubs the future
  runtime-loading surface" / "Everything below the core include is a
  placeholder to be filled in" ‚Äî all wildly out of date.
- `kashi_register_font` return-doc said `>= 2` ‚Äî should be
  `>= KASHI_RT_FONT_BASE` (which is 3 since 0.5.1).

**Fix**: all updated to reflect current reality.

### F4 (Info) ‚Äî `kashi_rt_glyph_row*` does two `_kashi_rt_get` lookups

`kashi_rt_glyph_row` and `kashi_rt_glyph_row_byte` each call
`_kashi_rt_get(font_id)` to read `RT_HEIGHT` / `RT_STRIDE`, then call
`kashi_rt_glyph_ptr(font_id, idx)` which calls `_kashi_rt_get` again. Two
`vec_get` lookups per row on the hot path.

**No change for 0.6.0** ‚Äî consumers wanting peak per-row speed should
hoist via `kashi_font_ptr(font_id, cp)` once and read rows with
`load8(ptr + row * stride + byte_idx)` directly (the documented hot-path
pattern from the guide). The bench `font_row_runtime ~42 ns` is the cost
of the convenience accessor and it's acceptable; optimizing it would
either inline the lookup (duplicating logic) or restructure the
accessors. Worth revisiting if a real consumer profile flags it.
