# Security & Hardening Audit — 2026-05-27

> **Scope**: the freestanding font-data core (`src/font_data.cyr`) and its
> public accessor surface, plus the library-skeleton (`src/lib.cyr`).
> **Trigger**: P(-1) hardening pass opening the post-0.1.0 cycle (CLAUDE.md
> § Process). **Auditor**: Claude (freestanding-core agent).
> **Toolchain**: cyrius 6.0.3.

## Summary

The freestanding accessor surface is **memory-safe for all `i64` inputs** —
no input to any caller-facing accessor can produce an out-of-buffer read or
a wild dereference. Three findings were actioned (one defensive fix, one
dead-state fix, one documentation-accuracy fix); two were noted without
code change. No glyph byte was modified — fidelity vs agnos source is
preserved (the fidelity tests still pass).

| # | Severity | Area | Status |
|---|---|---|---|

| F2 | Low | `kashi_font_ready` was write-only dead state | **Fixed** — `kashi_font_is_ready()` reader added |

## Bounds analysis (the core question)

Every caller-facing accessor was traced for the full signed-64-bit input
domain. All are total and safe:

- **`kashi_glyph_encoded(ch)`** — returns `1` only for `ch ∈ [0x20, 0x7F]`;
  `ch < 0x20` (incl. all negatives) → `0`, `ch > 0x7F` (incl. large
  positives) → `0`. The single gate every read path funnels through.
- **`kashi_glyph_ptr(font_id, ch)`** — gated by `kashi_glyph_encoded` and an
  explicit `font_id ∈ {0,1}` check. Max byte offset is `95*16+15 = 1535`
  (VGA, into the 1536-byte logical region) and `95*8+7 = 767` (CGA, into
  768). Unknown font or unencoded `ch` → `0` (null). No offset can exceed
  the buffer.
- **`kashi_glyph_row(font_id, ch, row)`** — encoded gate, then a height gate
  (`kashi_font_height` returns `0` for any unknown font → early `0`), then
  `row < 0 → 0` and `row >= h → 0`, then a null-check on the base pointer
  before `load8(base + row)`. `row` is therefore always in `[0, h-1]`, so
  the read offset never leaves the buffer.
- **Metadata accessors** (`width/height/first/count`) — return non-zero only
  for `font_id ∈ {0,1}`; any other id → `0`.

**Buffer sizing.** `var kashi_font16[1536]` / `kashi_font8[768]` at module
scope reserve 8× their byte footprint (12 288 / 6 144 bytes) under the
Cyrius u64-unit rule (see architecture note 001), addressed per-byte via
`store8`/`load8`. The 8× over-reservation — kept for parity with agnos's
`fb_font[1536]` — is an *accidental safety margin*: even a write at an 8×
miscomputed offset stays inside the allocation. Not relied upon by the
contract; not changed.

## Findings

### F1 — `kashi_fset16` / `kashi_fset8` wrote at an unchecked offset (Medium) — Fixed

The table-packing helpers computed `&buf + (ch - 0x20) * stride` and stored
without validating `ch`. They are documented internal and all current calls
pass `ch ∈ [0x20, 0x7F]` — but in a freestanding `include` every function is
callable, and, more importantly, **a typo in a future built-in font table**
(e.g. `kashi_fset16(0x1F, …)` or a transposed codepoint) would compute a
*negative* offset and write **before** the BSS buffer, silently corrupting
adjacent globals or faulting. With more built-in fonts booked for M2, the
table-authoring surface — and the typo risk — grows.

**Fix**: a two-line range guard at the top of each packer ignores any `ch`
outside `[FIRST, LAST]`. `kashi_font_init` runs once, so the cost is
negligible. No behavior change for correct (in-range) calls; no glyph byte
affected.

**Test**: `test_fset_guard` (src/test.cyr) fires a barrage of out-of-range
packs (below `0x20`, above `0x7F`, negative) at both fonts and asserts the
in-range table — and the ready flag — remain intact. (The suppressed write
is not directly observable from the language; the guard's primary assurance
is this analysis. With the current 8× over-reservation an *above*-range `ch`
lands harmlessly in padding, so the guard chiefly protects the
*below*-range / negative-offset case.)

### F2 — `kashi_font_ready` was write-only (Low) — Fixed

`kashi_font_init` set `kashi_font_ready = 1`, but no accessor ever read it —
dead exported state. A freestanding consumer (a kernel especially) has a
legitimate need to assert "did I call init?" before the first render.

**Fix**: added `kashi_font_is_ready()` (pure arithmetic; freestanding-safe —
`cyrius vet` still reports "no dependencies"). Pre-init access remains
bounds-safe: the buffers are zeroed BSS, so glyphs simply render blank.

**Test**: `test_font_ready` asserts the flag reads back set after init.

### F3 — Misleading "DEL=blank" comment on the VGA `0x7F` glyph (Low) — Fixed

The VGA 8×16 `0x7F` row comment read `# DEL=blank`, but the glyph data
(`10 38 6C C6 C6 C6 FE`) is the **CP437 triangle** glyph, not blank — it
matches IBM VGA / agnos source (the byte-fidelity tests confirm the bytes
are correct; only the comment was wrong). The CGA 8×8 `0x7F`, by contrast,
genuinely *is* blank.

**Fix**: corrected the inline comment on the VGA `0x7F` line and the
`KASHI_GLYPH_LAST` enum comment to state the per-font reality. No data
change.

### F4 — `kashi_glyph_row` recomputes the encoded/font gate (Info) — No change

`kashi_glyph_row` checks `encoded` and branches `font_id` (via
`kashi_font_height`), then calls `kashi_glyph_ptr`, which checks `encoded`
and branches `font_id` again — a few redundant branches on the
per-pixel-row hot path. At 17 ns/call (see `docs/benchmarks.md`) this is not
worth trading the current readability and the single-source bounds logic
for. Left as-is per "correctness over cleverness"; revisit only if a
profile of a real consumer shows the accessor on the critical path.

### F5 — Fuzz harness reaches only byte-range inputs (Low) — Mitigated

`tests/kashi.fcyr` derives `(font, ch, row)` from `load8`, so its effective
input domain is `[0, 255]³` — it never exercises negative or `> 255`
arguments, despite the header's "full int range" claim. The accessors *are*
safe across the full `i64` domain (see the bounds analysis), but the test
surface did not demonstrate it.

**Mitigation**: added `test_extreme_inputs` (src/test.cyr) pinning the
contract at the extremes — `0x7FFFFFFF` and large-negative `font_id`, `ch`,
and `row` all return the safe sentinels (`0` / null). The fuzz harness was
left byte-driven (reasonable for its random-stream model); the extremes are
now covered deterministically.

## Verification

- `cyaudit vet src/font_data.cyr` → `no dependencies` (boundary intact after
  the new accessor). NB: invoke `cyaudit` directly — the released 6.0.3
  `cyrius vet` dispatches to a `cybs` that emits an ELF instead of auditing
  (a cyrius packaging bug; CI and `docs/development/state.md` note the same
  workaround).
- `cyrius fmt --check`, `cyrius lint` → clean, 0 warnings.
- `cyrius test` → **96 assertions, 0 failed** (86 unit + 10 integration; up
  from 81 — the audit added 15).
- Glyph fidelity vs agnos source: unchanged (no byte touched).
